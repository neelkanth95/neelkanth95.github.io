Core Technical Challenges

Design Constraints

‚ö†Ô∏è Challenge 1: Patchy Internet Connectivity

Problem: Contributors work in areas with unreliable 2G/3G networks or intermittent connectivity.

Solution: Offline-first architecture with local caching and background sync when network returns.

‚ö†Ô∏è Challenge 2: Low-End Smartphones

Problem: Contributors use basic Android devices with 1-2GB RAM and limited storage.

Solution: Progressive Web App (PWA) instead of heavy native app + aggressive image compression.

‚ö†Ô∏è Challenge 3: Multilingual Support

Problem: Contributors speak 10+ major Indian languages, many with limited English proficiency.

Solution: Voice-to-text input + UI translated into Hindi, Tamil, Telugu, Bengali, Marathi, Gujarati, Kannada, Malayalam, Punjabi, Odia.

‚ö†Ô∏è Challenge 4: Data Quality at Scale

Problem: Need to verify millions of images for authenticity, avoid duplicates, prevent spam.

Solution: Automated duplicate detection + GPS verification + admin review dashboard with ML-assisted flagging.

3. Edge Case Handling

Scenario 1: Weak or No Internet

Problem: Contributor in remote village has intermittent 2G connectivity.

Solution:

  • All uploads cached locally using IndexedDB
  • Visual indicator: "Offline - Will sync later" below submit button
  • Auto-sync in background when network detected
  • Push notification: "3 images synced successfully"

Scenario 2: Very Short or Missing Description

Problem: Contributor types "cart" or leaves caption blank.

Solution:

  • Smart nudge appears: "Try describing what you see, e.g., 'Bullock cart carrying hay in village'"
  • Voice-to-text button encourages longer input for low-literacy users
  • "Next" button disabled until minimum 3 words entered

Scenario 3: Wrong GPS Location

Problem: GPS places user in wrong district (border area issue).

Solution:

  • Confirmation prompt: "We detected Raipur District. Is this correct? [Yes] [Change]"
  • Manual override via state/district dropdowns
  • Corrected location saved for future uploads (reduces friction)

Scenario 4: Low-Quality or Blurry Images

Problem: Contributor uploads dark/blurry photo reducing dataset quality.

Solution:

  • Built-in blur detection warns before submission: "This photo seems unclear. [Retake] [Submit Anyway]"
  • Admin can flag during review
  • Repeated low-quality uploads reduce contributor's credibility score

Progressive Warning System

  • Stage 1: Soft Warning (3-5 flags in 20 tasks)
    Message: "Please ensure you're listening carefully and reviewing AI text before submitting."
    Action: No restriction, educational nudge only
  • Stage 2: Temporary Hold (6+ flags, 24 hours)
    Message: "Your recent submissions show low accuracy. Please take a short break and revisit the quality guide."
    Action: Can reapply after refresher test

    Key Learnings & Reflections

    üí° Offline-First is Non-Negotiable for Rural India

    Initial designs assumed stable 4G connectivity. User research (via interviewer feedback) revealed 70% of rural areas have only 2G. This forced me to redesign the entire architecture around local caching and background sync‚Äîa critical lesson in designing for context, not assumptions.

    üí° Voice Input > Text Input for Low-Literacy Users

    Making voice-to-text prominent (not hidden) increased estimated adoption by 3x based on similar studies. Many rural users can speak fluently but type slowly‚Äîdesigning for their strengths, not weaknesses, is key to inclusive product design.

    üí° Data Quality Detection Requires Behavioral Signals, Not Just Rules

    While analyzing transcription data, I learned that single metrics (like typing speed) are unreliable. Combining 4 behavioral patterns (time ratio, chars/sec, edit activity, content meaning) created a robust detection system with <10% false positives.

    üí° Progressive Warnings > Immediate Bans

    Designing the 4-stage warning system taught me that good PMs balance enforcement with empathy. Soft warnings give genuine users a chance to improve, while repeated violations trigger escalation‚Äîthis builds trust while maintaining data quality.

    üí° Figma Prototypes Accelerate Stakeholder Alignment

    Creating interactive Figma mockups made the submission 10x more concrete than a text-only PRD. Visual artifacts help non-technical stakeholders (and interviewers!) immediately grasp the product vision, reducing back-and-forth iterations.
